# Team support / helpdesk system + embeddable widget

**Date:** 2026-06-14
**Status:** Approved

## Problem

Teams need a customer-support inbox: external visitors (and logged-in
coglionazzi users) start a chat, agents (team members) reply, triage by
**category**, and mark a ticket **resolved**. A drop-in `<script>` makes a
support bubble appear on any third-party site, opening a chat where a visitor
enters their email and starts talking to the team.

## Decisions (from brainstorming)

- **Requester = both.** External anonymous visitors (via the widget, identified
  by email + a per-ticket secret token) AND logged-in coglionazzi users (via an
  authenticated in-app entry point). The data model supports both; the widget is
  the primary deliverable.
- **Categories = team-configurable.** Each team defines its own.
- **Realtime = live (SSE)** for the widget; agents already get realtime via the
  existing chat WebSocket.
- **Widget = iframe-embedded.** A tiny loader script injects a bubble + an
  `<iframe>` pointing at a coglionazzi `/widget` page (full style isolation,
  reuses our React).

## Core architectural decision: reuse the existing message model

A support ticket's conversation is **not** a new message store. It reuses the
generic chat model (see CLAUDE.md → "Chat rooms & messages"):

- A new `chat_rooms` **kind `'support'`**, `owner_id = ticket.id`. Threads reuse
  `chat_messages`, `chat_message_reactions`, `chatPublisher`, `MessageThread`,
  and `useChatRoom`.
- **Sender convention in a support room:** `chat_messages.created_by IS NULL`
  ⇒ the visitor; a non-null user ⇒ an agent (team member). `created_by` is
  already nullable (`ON DELETE SET NULL`), so **no change to the messages
  schema**. The author join yields `null` for visitor messages; the agent UI
  renders that as "Customer", the widget renders non-null authors as the team.

Two access paths to the same room:

1. **Agents** — existing oRPC `chat.*`, with a new `support` branch in
   `roomAccess.ts` gating by the ticket's team membership. No new thread code.
2. **Visitors** — new **public** HTTP endpoints authed by the ticket's
   `access_token`, writing visitor messages (`created_by NULL`) and publishing
   to the same `chatPublisher(roomId)` so agents see them live; an SSE stream
   pushes agent replies back to the widget.

Rejected alternative: a separate `support_messages` table — duplicates the
composer, pagination, and realtime stack for no benefit.

## Data model (new migration)

`support_categories`
- `id uuid pk default gen_random_uuid()`
- `team_id uuid not null references teams(id) on delete cascade`
- `name text not null`
- `position int not null default 0`
- `created_at timestamptz not null default now()`
- `unique (team_id, name)`

`support_tickets`
- `id uuid pk default gen_random_uuid()`
- `team_id uuid not null references teams(id) on delete cascade`
- `subject text` (nullable — optional, agent-editable)
- `category_id uuid references support_categories(id) on delete set null`
- `status text not null default 'open'` — `check (status in ('open','resolved'))`
- `requester_user_id text references users(id) on delete set null` (when a
  logged-in coglionazzi user opened it)
- `requester_email text` (nullable; required for visitor tickets)
- `requester_name text` (nullable)
- `access_token text not null unique` — per-ticket visitor secret
  (`randomBytes(32).toString('hex')`)
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`
- `last_message_at timestamptz not null default now()` — inbox sort key
- `resolved_at timestamptz`
- index `(team_id, status, last_message_at desc)` for the inbox query

`teams` — add `widget_key text unique` (nullable). Public embeddable id (like an
Intercom app_id), lazily generated by `support.enableWidget` (owner-only).

Messages: `chat_messages` via the `{ scope: 'support', ticketId }` room ref.

### roomAccess.ts changes

- `roomRefSchema`: add `z.object({ scope: z.literal("support"), ticketId: z.uuid() })`.
- `assertRefAccess` / `resolveRoom` / `assertRoomAccess`: add a `support` branch
  — resolve ticket → `team_id` → `assertTeamMember`. (kind `'support'`,
  `owner_id = ticket.id`.)
- Cleanup: deleting a team must drop its tickets' rooms (extend `deleteTeamRooms`
  to include `kind='support'` for the team's ticket ids; ticket rows cascade off
  `teams`).

## Agent side (inside the app)

New oRPC router `src/server/orpc/support.ts` (`support.*`), team-gated via
`teamAccess`:

- `enableWidget({teamId})` (owner) → sets `widget_key` if null, returns it;
  `widgetKey({teamId})` → current key (member).
- `categories.list/create/rename/reorder` (member) / `delete` (owner).
- `tickets.list({teamId, status?, categoryId?, q?})` → tickets with category +
  requester + `messageCount` + `last_message_at`, newest-active first.
- `tickets.get({ticketId})` → full ticket (member).
- `tickets.createTicket({teamId, subject?, categoryId?, message})` (authP) — the
  in-app requester path: `requester_user_id = caller`, requester_email/name from
  the user; creates room + first message.
- `tickets.setStatus({ticketId, status})`, `tickets.setCategory({ticketId, categoryId})`,
  `tickets.setSubject` (member) — set `resolved_at` when resolving.
- `subscribe({teamId})` — yields `{teamId}` change signals (new shared `support`
  channel; see Realtime).

UI:
- A **"Support" space** in `<TeamPanel>` (sibling of Chat/Archive), route
  `routes/home/teams/$teamId/support.tsx`. Left: an inbox list (status +
  category + text filters, unread/active first). Right: the selected ticket —
  `<MessageThread roomRef={{scope:'support', ticketId}}>` plus a header with the
  status toggle, category `<Select>`, subject, and requester (email/name).
- Category management added to `TeamDialog` (the existing gear dialog).
- Inbox liveness: `support.subscribe` drives signal-and-refetch of
  `tickets.list` (a ticket's own thread is already live via `useChatRoom`).

### Realtime

New **shared `support` channel** on the existing `publisher`
(`server/realtime/publisher.ts`), mirroring `team`: `publishSupportChanged(teamId)`
called after ticket create / status / category / new message. `support.subscribe`
filters the shared channel by `teamId` (membership resolved once at subscribe).
Message-level liveness inside an open thread reuses `chatPublisher(roomId)`
unchanged — including messages posted via the **public** endpoints (they publish
to `chatPublisher` too, so agents' open threads update live).

## Public widget

### `public/widget.js` (hand-written vanilla loader)

Host site embeds:
```html
<script src="https://coglionazzi.com/widget.js" data-widget-key="KEY"></script>
```
It injects a floating bubble button + a hidden `<iframe src="https://coglionazzi.com/widget?key=KEY">`,
toggles open/closed, and relays unread counts / open-state via `postMessage`.
Static file (no Vite bundling needed); cacheable.

### `/widget` route (the iframe content)

A React route (`routes/widget.tsx`) with **no auth guard** and minimal chrome,
rendering the chat panel:
1. **Start form** — email (required) + name + category select (from
   `…/config`) + first message.
2. **Live thread** — visitor + agent messages, a plain-text composer, live via
   SSE.

Because the iframe is **same-origin** with the API, its `/api/support/*` calls
need no CORS. The route must be embeddable: serve it with
`Content-Security-Policy: frame-ancestors *` and **no** `X-Frame-Options: DENY`.
The widget persists its `accessToken` in the iframe's `localStorage` (keyed by
the coglionazzi origin) to resume a returning visitor's ticket.

### Public API (`src/routes/api/support/`)

TanStack Start server routes (the `createFileRoute(...).server.handlers`
pattern). `widget.js` is the only genuinely cross-origin caller (the iframe is
same-origin), but these endpoints set permissive CORS headers
(`Access-Control-Allow-Origin: *`, no credentials) to be safe and to allow a
non-iframe integration later:

- `GET …/config?widgetKey=` → `{ teamName, categories: [{id,name}] }`. 404 on
  unknown/disabled key.
- `POST …/tickets` `{widgetKey, email, name?, categoryId?, message}` → validates
  key → resolves team → creates ticket (`status open`, requester visitor) + room
  + first message (visitor, `created_by NULL`, text wrapped into minimal Lexical)
  → `publishSupportChanged(teamId)` → returns `{ticketId, accessToken, messages}`.
- `GET …/messages?token=` → ticket messages (authed by `access_token`).
- `POST …/messages` `{token, message}` → append visitor message; publish to
  `chatPublisher(roomId)` + `publishSupportChanged(teamId)`; bump
  `last_message_at`.
- `GET …/stream?token=` → **SSE**: resolve ticket → room → subscribe
  `chatPublisher(roomId)`, write `data: <json>\n\n` frames for `created`/`updated`/
  `deleted`. A `ReadableStream` bridges the async generator; closes on request
  abort.

Visitor text → Lexical: add a server helper `plainTextToLexical(text)` (inverse
of the existing `extractLexicalText`). The widget renders agent replies from
`body_text` (plain) for v1.

## Testing (vitest, real Postgres)

- Ticket creation: visitor path (via the public handler logic) and `createTicket`
  (authP) both create a room + first message with the right `created_by`.
- Token gating: `…/messages` and `…/stream` reject a wrong/unknown token; a valid
  token only reaches its own ticket.
- Widget-key resolution: `…/config` and `…/tickets` reject unknown/disabled keys;
  a valid key resolves the right team.
- Message append both directions: visitor append (`created_by NULL`) and agent
  reply (`chat.send`, non-null) land in the same room and publish to
  `chatPublisher`.
- Status/category: `setStatus` toggles + sets `resolved_at`; `setCategory`
  validates the category belongs to the ticket's team.
- Category CRUD + team-membership gating (non-member gets FORBIDDEN/NOT_FOUND on
  `tickets.list`/`get`, category mutations, `enableWidget`).
- SSE/publisher wiring is exercised by driving the generator / publisher
  directly (HTTP streaming itself is integration-level; not unit-tested).

Pure helpers (`plainTextToLexical`) get a unit test next to the source. Follow
TDD per the project's testing conventions.

## Build order (one spec, phased plan)

**Phase 1 — agent-facing ticketing.** Migration (categories, tickets,
`teams.widget_key`), `roomAccess` `support` branch + `deleteTeamRooms` extension,
`support.*` router (categories + tickets + status/category + `createTicket` +
`subscribe`), the Support space UI + category management in `TeamDialog`, the
`support` realtime channel. Fully usable internally.

**Phase 2 — public widget.** `enableWidget`/widget key surfacing in team
settings, public `/api/support/*` handlers (config/tickets/messages/stream),
`plainTextToLexical`, the `/widget` route (+ embeddability headers), and
`public/widget.js`.

## Out of scope for v1 (YAGNI)

- Rich-text rendering / file attachments in the widget.
- Agent assignment, SLA timers, internal notes.
- Widget theming (colors, welcome copy) — uses team name only.
- Email notifications (visitor/agent).
- Spam rate-limiting on `POST …/tickets` — noted as the first follow-up
  (the widget key is public, so unauthenticated ticket creation is abusable).
- Horizontal scaling: SSE + the support channel live in one process, same
  single-instance constraint as all existing realtime.
